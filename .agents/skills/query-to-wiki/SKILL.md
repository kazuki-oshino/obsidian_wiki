# Query To Wiki

このスキルは、既存 `wiki/` を読んで回答し、その回答のうち再利用価値が高いものだけを `wiki/` に還元するための定型ワークフローです。

## いつ使うか

- ユーザーが既存 `wiki/` の知識を整理、比較、概念化してほしいとき
- ユーザーが「wiki に残して」「今後の糧にして」「あとで見返せる形にして」のように durable な成果物を求めるとき
- `raw/*.md` の path は渡されておらず、既存 `wiki/` をもとに答えやノートを作るべきとき
- 回答後に、その内容を durable artifact として残すかどうかの判断まで含めて進めるべきとき

## 使わないとき

- ユーザーが `raw/*.md` の path を渡して wiki 化を求めているとき
  - この場合は `raw-to-wiki` を使う
- 一時的で再利用価値が弱い質問に、短い回答だけ返せば十分なとき
- wiki 全体の health check や矛盾点検が主目的のとき
  - これは Query より Lint 寄りの作業として扱う

## 最初に読むもの

1. `CLAUDE.md`
2. `wiki/index.md`
3. 関連しそうな既存 `wiki/*.md`
4. 必要なら、それらの `sources` から辿れる `raw/*.md`

`CLAUDE.md` の規約が最優先です。特に次を守ります。

- 回答は既存 `wiki/` を入口に組み立てる
- まず回答品質を優先し、その後に還元価値を判断する
- 更新は全面再生成ではなく delta update を基本にする
- 新規ノートより既存ノート更新を優先する
- ノートは原則日本語で書く

## 入力

- 必須: ユーザーの質問、比較依頼、整理したいテーマ
- 任意: 「既存ノートへ追記したい」「比較ノートとして残したい」「overview にしたい」などの追加指示

入力に `raw/*.md` が含まれる場合は、Query 還元だけでなく ingest が必要かどうかを先に切り分けます。

## 基本フロー

1. `wiki/index.md` と関連ノートを読み、まず回答内容を組み立てる
2. その回答が後からも再利用されるかを判断する
3. 再利用価値が高いときだけ、既存ノートの追記または新規ノート作成を行う
4. 新規ノートや入口の変化がある場合だけ `wiki/index.md` を更新する
5. ユーザーには、回答本体と wiki への反映内容を分けて伝える

実際の編集が回答前後どちらになる場合でも、判断の順序は常に `回答を作る -> 還元価値を判断する -> 必要時のみ更新する` とします。

## 還元判断

### 既存ノートへの追記を優先する

次の条件なら、まず既存ノートへの追記を検討します。

- すでに自然な受け皿になる `concept`、`overview`、`comparison` がある
- 既存ノートに判断基準、比較軸、運用上の注意を足すだけで十分
- 新規ノートを作ると重複や断片化が起きやすい

### 新しい `concept` を作るとき

次の条件を満たす場合だけ、新しい `concept` を検討します。

- 少なくとも 2 件以上の既存ノートを横断する安定した概念がある
- 既存ノートへの追記だけでは主題が埋もれる
- 今後もこのテーマを入口として参照する可能性が高い

### `comparison` を作るとき

次の条件を満たす場合だけ `comparison` を作成または更新します。

- ユーザーが比較を明示的に依頼した
- 既存ノート間に安定した比較軸、対立点、選択基準がある
- 後から見返す価値が高い

### `overview` を作るとき

次の条件を満たす場合だけ `overview` を作成または更新します。

- 関連ノートが増え、入口ページがある方が探索しやすい
- ユーザーが全体像、読み順、地図を求めている
- 既存の `index` だけではテーマ入口として粗すぎる

### 何も作らないとき

次の条件なら、回答だけ返して wiki 更新を見送ります。

- 既存ノートにすでに同じ整理がある
- 今回の回答が一時的で、後から再利用する可能性が低い
- 1 行のメモを増やすだけでノイズが増える
- 比較軸や概念がまだ安定していない

## Query 由来ノートの扱い

- `source-summary` は Query 還元の主軸にしない
- 新規ノートや大きな追記を行うときは `sources`, `updated`, `tags` を整える
- `sources` には、今回の根拠として直接参照した既存 `wiki/` ノートを入れる
- 必要に応じて、その場で確認した `raw/` ノートも `sources` に含めてよい
- どのノートを読んで整理したかが後から辿れる状態を保つ

## `wiki/index.md` 更新条件

次のいずれかに当てはまる場合だけ更新します。

- 新しい `concept`、`comparison`、`overview` を作った
- 既存ノートの役割や説明が変わり、入口情報も変えるべきになった

単なる軽微な追記だけなら、毎回 `index.md` を触る必要はありません。

## 避けること

- 毎回の Q&A を機械的に wiki 化すること
- 既存ノートに追記できるのに新規ノートを量産すること
- `raw-to-wiki` が担う ingest 作業までこのスキルで引き受けること
- 既存 `wiki/` を読まずに、会話の勢いだけで新規ノートを作ること
- 一時的なメモを durable artifact として残しすぎること

## 実行チェックリスト

- [ ] `CLAUDE.md`、`wiki/index.md`、関連 `wiki` を読んだ
- [ ] まず回答内容を組み立てた
- [ ] 再利用価値の有無を判断した
- [ ] 既存ノート更新を新規ノート作成より先に検討した
- [ ] 必要なら `concept` / `comparison` / `overview` を作成または更新した
- [ ] 必要な場合だけ `wiki/index.md` を更新した
- [ ] `sources`, `updated`, `tags` が辿れる形になっている
- [ ] 見送るべきケースで無理に artifact を増やしていない

## ユーザーへの返し方

作業後は次を簡潔に伝えます。

1. ユーザーの質問に対する回答
2. 作成または更新した `wiki` ノート
3. wiki 更新を見送った場合は、その理由
4. `wiki/index.md` を更新したかどうか
5. テストや確認をしていない場合はその旨
